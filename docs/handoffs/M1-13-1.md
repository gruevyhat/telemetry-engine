## Decisions + Spec basis

1. **The M1 acceptance bar requires the by-hand half too, not just headless** — Spec §21.3: "the
   solo trade campaign completes 4 turns headless AND by hand." Before this session, `App.tsx`
   only ran the fixed M0 demo script; no card covered wiring real content into it. This gap was
   found, carded (M1-13), and closed.
2. **M1-13**: wired the real trade-deck content into `App.tsx` — market feed via `marketAt`/
   `renderFeed`, incident generation via the phase interpreter's `"generate"` step, funds via
   `purchase.settled`/`sale.settled`. New content: `content/frames/trade-campaign/{turn.json,
   announce-templates.json}` (a 4-turn phase script). Basis: Spec §21.3, plus reusing exactly the
   projection/render functions already built and tested in earlier M1 tasks (M1-01–M1-12) rather
   than inventing new ones.
3. **M1-14**: added a real skill-check step (TRANSIT beat, `skillSlot: "pilot"`, difficulty 7)
   using the interpreter's pre-existing `"check"` step kind — no interpreter change needed, since
   that machinery already existed and was just never exercised by the live script. Basis: Spec §6
   ("never rolls for the player") — the UI is a roll-entry control, not an auto-roller.
4. **M1-15 (owner-approved, see card)**: added `PhaseInterpreter.reportCheck(t, actor, {skill, dm,
   total, difficulty})` as a new mid-beat interpreter action API, because NPC interrogation is
   player-initiated inside a beat (not a beat transition) and INV-6 ("nothing writes to the ledger
   except the phase-engine interpreter") means only the interpreter may append the resulting fact.
   Two designs both fit the Spec's silence here (script-encoded sub-sequence vs. new interpreter
   action method); the implementer stopped and asked per CLAUDE.md's "Spec is silent and two
   materially different implementations both fit" rule. Owner approved option 2, recorded in
   `docs/tasks/M1-15.md`'s "Owner decision (2026-07-18)" section — read it, it's the primary
   source, not this summary.
5. **M1-16 (owner-approved, see card)**: evidence reveal wired at COMMS using Spec §10.1's
   `FactSelector`/access-gate/Effect-ranked-reveal machinery already built (and untouched) in
   `packages/engine/src/evidence/evidence.ts` since M1-07. This one explicitly touches Visibility
   (a reveal fact widens a target fact's effective scope toward `table`), which CLAUDE.md's hard
   rule gates behind "unless your task card names it" — so it needed sign-off before starting, not
   just before merging. Recorded in `docs/tasks/M1-16.md`'s "Owner decision (2026-07-18), corrected
   after further research" section.

## IMPORTANT — a correction worth reading before touching `phases/` again

While researching M1-16 ("does an interpreter action already exist for this"), the implementer
discovered `packages/engine/src/phases/commits.ts` already contained `commitInterrogationAnswer`
and `commitEvidenceReveal` — fully built, fully tested standalone functions solving exactly the
INV-6 mid-beat-action problem M1-15 and M1-16 were wrestling with, with **zero callers anywhere in
the codebase** before this session. Consequences:

- **M1-15's first implementation was incomplete.** It added `reportCheck` (genuinely needed and
  correct — nothing committed `check.reported` before it either) but the interrogation *answer*
  itself was never committed to the ledger as the real `npc.statement` (table-visibility) /
  `npc.truthTierAssigned` (referee-visibility) fact pair `fact-kinds-v0.md` §3 defines — it only
  lived in React state as presentation text. Found and fixed in a follow-up commit (`219e11b`,
  `fix(ui-shared): M1-15's interrogation now commits npc.statement/npc.truthTierAssigned`) which
  added `runInterrogation()` (exported from `App.tsx`) calling `commitInterrogationAnswer`.
- **M1-16's plan was corrected before implementation** to *not* add a new `resolveEvidence`
  interpreter method (the original text of that card's owner decision), and instead wire together
  three pre-existing, untouched functions: `evaluateAccess` (pure, called before ever showing a
  roll control — this is what keeps "access failure narrates and stops, no roll, no day cost"
  honest) → `rankAndPlanReveal` → `commitEvidenceReveal`. `PhaseInterpreter` was not modified for
  M1-16 at all.
- **Lesson recorded in both task cards' Correction sections**: grep
  `packages/engine/src/phases/commits.ts` before concluding no ledger-write path exists for a
  player-initiated action. Do this again before adding any further mid-beat action API.
