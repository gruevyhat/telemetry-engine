## The actual gap (why bullet 3 and the demo doc are blocked)

`packages/sim/src/campaign.ts`'s `runSocialCampaign` layers bot decision-making (`decideSocial`) on top of the **M1** incident loop; its own docstring says a scripted turn "would step into confrontation/comms machinery M1 doesn't have." It never calls `dealAgendas`, `queueCommsAction`, `resolveConfrontation`, or the black-box functions. `metrics.ts` already has the misattribution/detection-rate machinery *waiting* for an M2-aware caller ("M1 has no confrontation/vote... an M2 caller wires `accused`... against that") — but nothing wires it yet.

Separately, no part of the codebase has ever assembled M2-08 through M2-14's standalone, unit-tested components into a runnable app: `packages/ui-shared/src/App.tsx` is still the M1 single-device hotseat demo (no pairing, no phone split); `packages/ui-phone/src/App.tsx` is a one-line stub; there is no session orchestrator; there is no production agenda-deck JSON under `content/` (only inline test fixtures exist, e.g. this session's `DECK` in `social-campaign.integration.test.ts`). A real "pair at least two phones to the local shared-screen host... finish by verifying the black-box preimages" walkthrough (Plan §5's M2 exit demo) needs all of that assembled — host+phone+transport+content — which is its own multi-concern effort well past this card's ~600-line budget, not a natural extension of the tests above.

I did not build through this. Per CLAUDE.md's stop condition (diff budget / scope needing owner sign-off) and because the milestone-close gate is explicitly owner-manual (Plan §6.2: "the owner runs the demo... fun is assessed"), this needs the owner's decision, not mine.
