## The blocker (where I got stuck)
Every `.tsx` test file in `packages/ui-shared/src/shared-screen/` fails identically:
```
Error: Failed to resolve import "react/jsx-dev-runtime" from "...Ticker.test.tsx".
Plugin: vite:import-analysis
```
This happens whether or not the test uses the `// @vitest-environment jsdom` pragma (error phrasing differs slightly — "resolve" returns null with the pragma vs "load" fails without it — but both fail).

**Confirmed NOT the cause:**
- `react` is genuinely installed and linked at `packages/ui-shared/node_modules/react`, including `jsx-dev-runtime.js` with a correct `exports` map entardy.
- Node's own `require.resolve('react/jsx-dev-runtime', {paths: [...]})` from that same directory finds it instantly — this is Vite-specific, not a real missing-package problem.
- Not a stale-cache issue (cleared `node_modules/.vite`, no change).
- Not `resolve.dedupe` (tried `["react", "react-dom"]`, no change).
- Not `resolve.conditions` (tried `["node", "import", "default"]`, no change).
- Not `server.fs.allow` (tried allowing `..`, no change).
- Not `test.server.deps.inline` (tried inlining `/react/` and `/@testing-library/`, no change).

**Debug evidence:** `DEBUG=vite:resolve-details` shows `[fallthrough] react/jsx-dev-runtime` immediately followed by resolution returning `null` — i.e. every one of Vite's `resolveId` hooks (including `@vitejs/plugin-react`'s) declines to handle it, and Vite's own default node-resolve fallback also fails, even though (per above) plain Node resolution from the same directory succeeds. `DEBUG=vite:config` shows the active resolve config has `mainFields: []` and `conditions: ['node']` — this may just be vitest's normal SSR-style test config (unconfirmed whether that's expected or the actual culprit).

**Untried idea, worth trying first in the next session:** the vitest config's `esbuild` block already shows `{ jsxDev: true, jsx: 'automatic', target: 'node18' }` — meaning vitest/esbuild can already do the automatic-JSX-runtime transform *without* `@vitejs/plugin-react` at all (that plugin's main value-add is Babel-based Fast Refresh for dev servers, irrelevant to running tests). **Try removing `@vitejs/plugin-react` from `vitest.config.ts` entirely** and let esbuild's native automatic-runtime transform handle `.tsx` files — it's possible the plugin's Babel transform is injecting the `react/jsx-dev-runtime` import in a form/timing that Vite's resolver then can't handle correctly in this pnpm-symlinked monorepo layout, and esbuild's own transform (already active per the config dump) may resolve it differently and correctly.

Other untried ideas if that doesn't work: try running vitest with `--pool=forks` instead of the default; try invoking vitest from inside `packages/ui-shared` with its own dedicated config instead of one shared root config; check whether `pnpm config get node-linker` is `isolated` (default) and whether switching it changes anything (last resort, would need `pnpm install` again); search for this exact error text combined with "pnpm workspace" — it may be a known vitest/vite issue with a documented fix.

## Decisions made so far (Spec basis, for the eventual PR/commit message)
1. **te-wireframe-shared.svg doesn't exist in the repo** — built the four regions from the rulebook §3.1 prose description instead ("Status bar (top)... Phase track... Main panel (center)... Ship's log ticker (bottom)").
2. **PhaseTrack shows exactly 4 beats**, not all 5 `BeatSlot` values — confirmed via rulebook §6 ("Each turn is one ship-week in four beats... Beat 1 DOCKSIDE, Beat 2 COMMS WINDOW, Beat 3 TRANSIT, Beat 4 ARRIVAL"). DOWNTIME exists as a `BeatSlot` (M0-02) but isn't part of the turn's phase track.
3. **StatusBar's funds/obligation are meant to be wired to the real M0-03 projections** (`fundsProjection`, `clocksProjection`) — the test constructs a real ledger and derives both via `derive()` to prove the pipeline works end-to-end. `hex`/`fuelTons`/`holdState` have no projection yet (no fact kind cleanly derives ship location or remaining fuel at v0 — `jump.plotted`/`fuel.consumed` exist but a full derivation wasn't attempted, out of scope) so they're plain props for now.

This mirrors the "impossible by construction" pattern used for split-visibility payloads in M0-02.
5. **Dependencies added** (owner approved adding jsdom + `@testing-library/react` before this work started — see conversation): `jsdom`, `@vitejs/plugin-react` (root, for `.tsx` test support); `@testing-library/react` (ui-shared).
