## What the session orchestrator actually is (piece 3) — read this before extending it

`createSocialSession(deps: { interpreter: PhaseInterpreter })` is a **pure translation layer**, not a running session. It has four methods:

- `handleCommsQueue(t, message: ProtocolPayloadMap["comms.queue"])` → calls `interpreter.queueCommsAction`, returns a `comms.ack` payload.
- `closeCommsWindow(t, actor)` → calls `interpreter.advance`.
- `castVote(topic, playerId, value)` → accumulates one ballot into an in-memory `Map<topic, Record<playerId, boolean>>`. No ledger write.
- `closeConfrontation(input)` → reads the accumulated ballots for `input.topic`, calls `interpreter.resolveConfrontation` once, and shapes the result into a `vote.resolved` payload (`status`/`outcome`) plus the raw `committed` facts.

The test (`social-session.test.ts`) proves the whole thing by round-tripping every message through **real** `encryptMessage`/`decryptMessage` (AES-GCM, same as production) before it ever reaches the orchestrator — that's what "not direct interpreter calls" means. It does not use `@telemetry/transport-webrtc`'s trystero adapter or `PairingHost`/`PairingClient` at all; it uses one shared symmetric key and constructs `BoundHeader`s by hand. That's a deliberate scope cut to keep this piece a normal red/green unit, not a scope gap to paper over: **wiring this orchestrator to a real `PairingHost` (per-player keys, `snapshot()` for outbound `state.snapshot`, the actual room/action layer) is exactly the remaining app-glue work in piece 4**, not something piece 3 already covers.

Two design choices worth knowing before extending this:

- **Ballot accumulation is single-shot, not incremental-resolve.** The wire protocol casts one ballot per `vote.cast` message, but `resolveConfrontation` takes a full `ballots` map and commits a fresh `vote.recorded` fact (plus terminal effects if it carries/fails) on every call. Screens-v2 §4.2 implies each cast should append its own public tally fact — i.e., `resolveConfrontation` called once per cast, incrementally. This orchestrator does **not** do that; it collects all casts in memory first and calls `resolveConfrontation` exactly once when the caller decides voting is done (there's no "close the vote" trigger wired to anything — that's the caller's job, same as `closeCommsWindow` is a separate explicit call from every `castVote`/`handleCommsQueue`). If piece 4 needs the real per-cast public tally behavior, that's a small but real design gap between the wire protocol's shape and `resolveConfrontation`'s one-shot API — flag it to the owner rather than guessing at idempotent multi-call semantics for `resolveConfrontation`, since calling it twice with a growing ballot set was not verified safe (it could double-append terminal effects like `envelope.opened` if called again after already carrying).
- **`agenda.actionTaken` unblocked without a `target` field.** The test's fixture `AgendaActionContent` omits the optional `target` field entirely (unlike `content/decks/trade/agendas.json`'s real actions) because its `proposals` use a literal payload value rather than `{ref: "target", field: ...}`. If piece 4 wires the orchestrator against the real trade-deck agendas, `targetFactId` resolution against `target.kinds` needs the real content shape, not this test's minimal one.
