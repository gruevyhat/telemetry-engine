# M1-16 — Evidence reveal UI

## Context

`docs/design/screens-v1.md` §2.3 designed this screen for M1; M1-07 built the engine-level
`FactSelector` evaluation, access gate, Effect-ranked reveal, and atomic day-cost transaction
(`packages/engine/src/evidence/evidence.ts`), proven by its own property/unit suite. No card ever
wired it into `App.tsx`; M1-13 scoped it out explicitly.

Same INV-6 "player-initiated mid-beat action" problem as M1-15 (see that card — a target selector
+ access precondition + check, chosen live by the player, has no interpreter API to commit through
today). This card additionally, and more centrally, **touches Visibility**: an evidence reveal
widens a fact's scope from wherever it currently sits (commonly `referee`) toward `table` via a
`reveal` fact — that is the literal mechanism `docs/design/screens-v1.md` §2.3 describes ("REVEALED
(widened to table)"). CLAUDE.md's hard rule is explicit: "Do not touch `Visibility` handling unless
your task card names it." This card names it, which is exactly why it needs owner sign-off before
implementation, not a green light to proceed unsupervised — the invariants at stake (INV-11's
atomicity, and Visibility's per-fact scope model generally) are ones the Spec treats as
load-bearing enough to warrant a standing "ask first" rule.

## Owner decision (2026-07-18)

Approved, following M1-15's pattern: a new interpreter action, `PhaseInterpreter.resolveEvidence(t,
actor, query, candidateFacts, effect, context?)`. It calls the existing, already-tested
`rankAndPlanReveal` (`packages/engine/src/evidence/evidence.ts`) to get an `EvidencePlan` — that
function is not touched, not re-derived, its atomicity guarantee is reused verbatim. If the plan is
`ok`, the new action commits `plan.revealProposals` via `ledger.appendAll` (already atomic per
INV-11: validates every input before appending any). If access failed, nothing is committed and the
denial reason is returned for the UI to narrate — matching Spec §10.1's "access failure narrates
and stops, no roll, no day cost." The access check itself (`evaluateAccess`) is pure/read-only, so
the UI calls it directly (no ledger involved) *before* ever showing a roll-entry control, keeping
the no-roll-on-denial ordering explicit rather than relying on `rankAndPlanReveal`'s internal check
alone.

This is the actual Visibility-touching part CLAUDE.md's hard rule is gated on: `reveal` facts don't
mutate the target fact's own `visibility` field (facts are append-only, INV-2) — `ledger.visibleTo`
only ever reads a fact's own static `visibility`. A `reveal` fact is a new, separately `public`
fact recording that certain fields of a target fact are now common knowledge; a renderer that wants
to *show* a normally-hidden fact's content still has to cross-reference committed `reveal` facts
itself. This UI's job is to commit that reveal fact correctly and render its own recorded fields —
not to build a general "look up what's revealed" renderer, which is out of scope here.

## Tests first
- `interpreter.test.ts`: `resolveEvidence` with an access-denied context commits nothing (ledger
  unchanged) and returns the denial reason; with access granted, commits exactly
  `plan.revealProposals` (reveal fact(s) + the atomic day-cost `clock.tick`) via one `appendAll`
  call, matching `rankAndPlanReveal`'s own already-tested output for the same inputs
- `App.test.tsx`: an "Investigate" control at DOCKSIDE is denied (no roll shown) when access fails,
  and on access-granted + a submitted roll, the ship's log gains a `reveal` fact and a `clock.tick`,
  and the UI renders which fields were revealed — identity (`actor`) only appears after all
  non-identity fields are exhausted, matching the query's `identityFields` ordering

## Done when
- `PhaseInterpreter` gains `resolveEvidence` (engine change), covered by the tests above
- `App.tsx` renders an evidence-reveal control (reachable at DOCKSIDE, reusing that turn's
  incident's own cause facts as `candidateFacts` — the same facts source M1-15's interrogation
  control reads) built against `evaluateAccess`/`rankAndPlanReveal`, not a reimplementation

## Do not
- Do not build this without the owner decision above recorded in this card first
- Access failure narrates and stops — no roll, no day cost (Spec §10.1); the UI must not let a
  failed-access attempt reach the check step at all
- Do not weaken or bypass `evidence.ts`'s existing atomicity test coverage while wiring it up
- Identity-last per the fact-kinds catalog's rule — do not let a reveal expose an actor's name
  before it exposes surrounding circumstance
