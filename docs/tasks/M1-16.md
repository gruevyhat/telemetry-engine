# M1-16 ‚Äî Evidence reveal UI

## Context

`docs/design/screens-v1.md` ¬ß2.3 designed this screen for M1; M1-07 built the engine-level
`FactSelector` evaluation, access gate, Effect-ranked reveal, and atomic day-cost transaction
(`packages/engine/src/evidence/evidence.ts`), proven by its own property/unit suite. No card ever
wired it into `App.tsx`; M1-13 scoped it out explicitly.

Same INV-6 "player-initiated mid-beat action" problem as M1-15 (see that card ‚Äî a target selector
+ access precondition + check, chosen live by the player, has no interpreter API to commit through
today). This card additionally, and more centrally, **touches Visibility**: an evidence reveal
widens a fact's scope from wherever it currently sits (commonly `referee`) toward `table` via a
`reveal` fact ‚Äî that is the literal mechanism `docs/design/screens-v1.md` ¬ß2.3 describes ("REVEALED
(widened to table)"). CLAUDE.md's hard rule is explicit: "Do not touch `Visibility` handling unless
your task card names it." This card names it, which is exactly why it needs owner sign-off before
implementation, not a green light to proceed unsupervised ‚Äî the invariants at stake (INV-11's
atomicity, and Visibility's per-fact scope model generally) are ones the Spec treats as
load-bearing enough to warrant a standing "ask first" rule.

## Owner decision (2026-07-18), corrected after further research

Approved in principle: reuse `evidence.ts` rather than reimplement, and keep the commit path inside
`packages/engine/src/phases/` per INV-6. The original text of this decision proposed building a new
`PhaseInterpreter.resolveEvidence` method ‚Äî **superseded before implementation**: researching M1-16
surfaced that `packages/engine/src/phases/commits.ts` already ships `commitEvidenceReveal(ledger,
plan)`, an atomic `ledger.appendAll(plan.revealProposals)` wrapper, fully tested in
`evidence.test.ts`'s "commitEvidenceReveal ‚Äî atomic via ledger.appendAll [INV-11]" suite, with zero
callers anywhere in the codebase. (The same gap existed for M1-15's `commitInterrogationAnswer` ‚Äî
see that card's Correction section. Same file, same pattern, found once and applied to both.)

**No new interpreter method needed.** The full flow is three existing functions, none of them
touched: the UI calls `evaluateAccess` directly (pure, read-only, no ledger ‚Äî this is what keeps
"no roll, no day cost on denial" honest, ahead of ever showing a roll control) to decide whether to
show a roll-entry control at all; on submit, `rankAndPlanReveal(query, candidateFacts, effect, t,
context)` builds the `EvidencePlan`; if `plan.ok`, `commitEvidenceReveal(ledger, plan)` commits it
atomically. `PhaseInterpreter` is not modified.

This is still the Visibility-touching part CLAUDE.md's hard rule is gated on, and the sign-off
still stands: `reveal` facts don't mutate the target fact's own `visibility` field (facts are
append-only, INV-2) ‚Äî `ledger.visibleTo` only ever reads a fact's own static `visibility`. A
`reveal` fact is a new, separately `public` fact recording that certain fields of a target fact are
now common knowledge; a renderer that wants to *show* a normally-hidden fact's content still has to
cross-reference committed `reveal` facts itself. This UI's job is to commit that reveal fact
correctly and render its own recorded fields ‚Äî not to build a general "look up what's revealed"
renderer, which is out of scope here.

## Tests first
- `App.test.tsx`: an "Investigate" control at COMMS (not DOCKSIDE -- a turn's incident only
  commits its cause facts on the click that *leaves* DOCKSIDE, per M1-13's advance-resolves-
  fromStep timing, the same reason M1-15's interrogation control also lives at COMMS, not
  DOCKSIDE) is denied (no roll shown, nothing committed)
  when `evaluateAccess` fails; on access-granted + a submitted roll, `rankAndPlanReveal` ->
  `commitEvidenceReveal` lands a `reveal` fact and the atomic day-cost `clock.tick` in the ledger,
  and the UI renders which fields were revealed ‚Äî identity (`actor`) only appears after all
  non-identity fields are exhausted, matching the query's `identityFields` ordering. Prefer, like
  M1-15's `runInterrogation` fix, a standalone exported helper function over inlining the
  evaluateAccess/rankAndPlanReveal/commitEvidenceReveal sequence in a closure, so it's testable
  against a real ledger directly rather than only through the DOM

## Done when
- `App.tsx` renders an evidence-reveal control (reachable at COMMS, reusing that turn's incident's
  own cause facts as `candidateFacts` ‚Äî the same facts source M1-15's interrogation control reads)
  built against `evaluateAccess`/`rankAndPlanReveal`/`commitEvidenceReveal`, not a reimplementation
  and not a new `PhaseInterpreter` method

## Do not
- Do not build this without the owner decision above recorded in this card first
- Access failure narrates and stops ‚Äî no roll, no day cost (Spec ¬ß10.1); the UI must not let a
  failed-access attempt reach the check step at all
- Do not weaken or bypass `evidence.ts`'s existing atomicity test coverage while wiring it up
- Identity-last per the fact-kinds catalog's rule ‚Äî do not let a reveal expose an actor's name
  before it exposes surrounding circumstance
