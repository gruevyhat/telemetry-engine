# BL-03 — Type-check the INV-6 ledger-write lint rule instead of name-matching it

## Context
`eslint-rules/no-ledger-writes-outside-interpreter.js` was written in M0-01, before the real
`Ledger` type existed (M0-02). It is a v0 syntactic matcher: it flags `<ident>.append(...)` only
when the identifier is literally named `ledger` and the file isn't under `packages/engine/src/
phases/`. A rename (`const l = getLedger(); l.append(fact)`) or a function parameter
(`function commit(store) { store.append(fact); }`) slips past it undetected — INV-6 (only the
phase-engine interpreter writes to the ledger) is currently enforced by variable-naming
convention, not by anything that actually knows what a `Ledger` is. Flagged in the M0 retro
(`docs/retros/M0.md`, Spec health #7) — the real `Ledger` type has existed since M0-02 and nobody
has revisited the rule since.

## Tests first
- `eslint-rules/__tests__/no-ledger-writes-outside-interpreter.test.js` gains cases where the
  receiver is provably typed `Ledger` under a different variable name (e.g. via a type-checked
  ESLint rule using `@typescript-eslint/utils`'s parser services, or an equivalent typed-AST
  approach) and the call is still flagged outside `phases/`; and a case where a same-named
  variable that is *not* a `Ledger` (e.g. a plain object with an unrelated `.append` method) is
  correctly left alone — the current rule cannot make that distinction and a fix must.

## Done when
- the rule flags a real `Ledger.append` call regardless of the variable's name, and does not flag
  an unrelated `.append` call on a same-named non-`Ledger` variable
- `pnpm lint` still passes across the existing codebase with no changes required outside the rule
  itself and its tests

## Do not
- Do not weaken the existing name-based cases in the current test suite — add to them, don't
  replace coverage
- Do not touch `Visibility` handling
- If a type-checked ESLint rule requires a new dependency (e.g. `@typescript-eslint/utils`,
  `@typescript-eslint/type-utils`) beyond what's already in the workspace, that is a "new
  dependency" stop condition per CLAUDE.md — flag and get sign-off before adding it
