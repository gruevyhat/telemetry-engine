# M1-11a — Content format v1 + trade deck (regenerates incident-cards-spec)

## Carried over from M1-10 (degradation ladder) — UPDATE: engine handoff already landed
M1-10 built `packages/engine/src/degrade/ladder.ts`'s `runDegradeLadder()` (the rung 1-4
mechanism, Spec §17) but didn't wire it in. That handoff has since landed directly
(`phases/interpreter.ts`'s `resolveStep`'s `"generate"` case now calls `runDegradeLadder()`
instead of throwing when a named frame is missing from the deck, committing `degrade.reported`
+ whichever rung's real content actually fired) — it did **not** need a full composer first:
`generate` steps are frame-by-id (`phase-script.schema.json`'s `gen: {frameId}` has no
pool-reference shape), so "composer exhaustion" for this milestone means exactly "the named
frame is unavailable," which is already a real, reachable condition today. Building a full
weighted-eligible-pool composer (`generate/frame.ts`'s `eligibleFrames`/`CooldownState`) just to
have something to exhaust would be gold-plating ahead of the real content this task authors —
skip it unless the trade deck's actual shape turns out to need one.
What remains genuinely open for this task: no generic-family frame deck exists yet (M1-11b), so
rung 1 always falls through to rung 2 (the oracle) in practice until that content lands — that's
expected, not a bug to fix here.
