# M2-05 — Comms-window queue, seeded close-order, and fizzles

## Tests first
- property: enqueue timing never affects outcome; only the seeded `comms-order` permutation does
- property: restart before close produces the same order and facts as uninterrupted resolution
- integration: two actions targeting one resource resolve sequentially; the later one becomes referee-scoped `action.fizzled`

## Done when
- the `commsWindow` step atomically closes a durable/idempotent batch, validates each action against prior batch results, and transitions once
- failed actors receive private feedback in their next window without exposing timing or identity to others
- any new queue fact kind lands in `docs/design/fact-kinds-v0.md` before code, per M2-00's owner decision

## Do not
- no timestamp/tap-speed ordering and no live effects before close
- do not leave the queue only in interpreter-lifetime memory
