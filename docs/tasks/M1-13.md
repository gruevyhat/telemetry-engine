# M1-13 ‚Äî Wire the real trade campaign into the playable UI

## Context

Every M1-01..M1-12 engine module (`market.ts`/`marketAt`, `generate/frame.ts`, `degrade/ladder.ts`,
`render/feed.ts`, `MarketFeed.tsx`, `createPhaseInterpreter`'s `"generate"` step) lands with its
own tests and is exercised headlessly by `packages/sim`'s `runCampaign` ‚Äî `pnpm sim:smoke` is
green. But `packages/ui-shared/src/App.tsx` still only runs the M0 scripted demo
(`content/frames/demo/turn.json`): one fixed lap of four literal-fact `announce` steps, never
`marketAt`, never a `"generate"` step, never `MarketFeed`. No M1 card wired the real modules into
the browser UI, so Spec ¬ß21.3's M1 acceptance line ‚Äî "solo trade campaign completes 4 turns
headless *and* by hand" ‚Äî is only true for the headless half. This card closes that gap. Found
and carded while confirming M1 was ready for its milestone PR; no code changed before this card
was written.

`createPhaseInterpreter` already supports everything this needs without interpreter changes: a
`"generate"` step draws from a supplied deck via `deps.deck`/`deps.rng` and falls back through the
degrade ladder (rung 1 generic frame, rung 2 oracle) when the named frame is unavailable or
exhausted (`packages/engine/src/phases/interpreter.ts`). `marketAt(facts, hex, day)` and
`renderFeed(...)` (feeding `MarketFeed`) are also already built and tested; they are simply never
called from `App.tsx`.

## Tests first
- `App.test.tsx`: a play-through test drives four full turns (16 `advance()` calls across
  DOCKSIDE/COMMS/TRANSIT/ARRIVAL) against the real trade deck and asserts: the dockside step
  renders `MarketFeed` lines derived from `marketAt`/`renderFeed` (not literal fixture text); at
  least one turn's DOCKSIDE step is a `"generate"` step that can fire a real trade-deck incident
  (assert on the resulting `rendered` text or committed fact kind, not on which specific frame
  drew ‚Äî the draw is seeded but content-agnostic to this test); funds change after a buy and a
  sell action, matching `fundsProjection` over the `purchase.settled`/`sale.settled` facts posted
- snapshot: the new 4-turn script's `announce`-step render keys resolve against real templates
  (extend M1-09's existing template-snapshot suite rather than duplicating it)

## Done when
- a new phase script (JSON, alongside `content/frames/demo/`) drives DOCKSIDE ‚Üí COMMS ‚Üí TRANSIT ‚Üí
  ARRIVAL for at least 4 turns, with each DOCKSIDE (or a clearly-documented equivalent beat) a
  `"generate"` step over `content/decks/trade/frames.json` (falling back to
  `content/decks/generic/frames.json` and the oracle per the existing ladder, unchanged)
- `App.tsx` runs this script via `createPhaseInterpreter(ledger, script, { rng, deck })` instead
  of the M0 demo script; the M0 demo script and its render remain intact and reachable only via
  its own test coverage ‚Äî do not delete `content/frames/demo/turn.json` or its tests
- the DOCKSIDE screen renders `MarketFeed` fed by `marketAt`/`renderFeed` real projections, with a
  minimal buy action (posts `purchase.settled`); ARRIVAL posts `sale.settled` from the real
  current price, replacing the M0 demo's fixed-amount literal fact
- `StatusBar`'s funds field reflects `fundsProjection` changing across the 4-turn playthrough
  (today it only ever shows `0` because no real settlement fact is ever posted in the UI)
- a person can open the built app and click through 4 turns to completion without a console error
  or an unplayable dead end (verify by hand per CLAUDE.md's UI rule, not just by test)

## Do not
- No interrogation-ladder or evidence-reveal overlay (`docs/design/screens-v1.md` ¬ß2.2/¬ß2.3) ‚Äî
  those are follow-on cards; a `"generate"` step firing an incident and rendering its stub text in
  the main panel is sufficient for this card
- No confrontation/comms-window resolution ‚Äî COMMS stays the existing pass-through stub
  (`commsWindow` step kind is already a no-op batch-resolution stub per M2)
- Do not swap the interpreter's `renderSurfaceStub` for M1-09's real `createTemplateRenderer` ‚Äî
  that is a separate, real gap (the interpreter still stubs `"generate"`/oracle rendering rather
  than calling the real renderer) but conflating the two risks blowing this card's diff past
  CLAUDE.md's ~600-line split point; flag it as a follow-on rather than fixing it here
- No new fact kinds ‚Äî `purchase.settled`/`sale.settled` already exist and are what
  `fundsProjection` reads; do not invent a `market.trade` kind
- Do not touch `Visibility` handling
