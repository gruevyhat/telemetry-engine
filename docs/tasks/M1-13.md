# M1-13 — Wire the real trade campaign into the playable UI

## Context

Every M1-01..M1-12 engine module (`market.ts`/`marketAt`, `generate/frame.ts`, `degrade/ladder.ts`,
`render/feed.ts`, `MarketFeed.tsx`, `createPhaseInterpreter`'s `"generate"` step) lands with its
own tests and is exercised headlessly by `packages/sim`'s `runCampaign` — `pnpm sim:smoke` is
green. But `packages/ui-shared/src/App.tsx` still only runs the M0 scripted demo
(`content/frames/demo/turn.json`): one fixed lap of four literal-fact `announce` steps, never
`marketAt`, never a `"generate"` step, never `MarketFeed`. No M1 card wired the real modules into
the browser UI, so Spec §21.3's M1 acceptance line — "solo trade campaign completes 4 turns
headless *and* by hand" — is only true for the headless half. This card closes that gap. Found
and carded while confirming M1 was ready for its milestone PR; no code changed before this card
was written.

`createPhaseInterpreter` already supports everything this needs without interpreter changes: a
`"generate"` step draws from a supplied deck via `deps.deck`/`deps.rng` and falls back through the
degrade ladder (rung 1 generic frame, rung 2 oracle) when the named frame is unavailable or
exhausted (`packages/engine/src/phases/interpreter.ts`). `marketAt(facts, hex, day)` and
`renderFeed(...)` (feeding `MarketFeed`) are also already built and tested; they are simply never
called from `App.tsx`.

## Tests first
- `App.test.tsx`: a play-through test drives four full turns (16 `advance()` calls across
  DOCKSIDE/COMMS/TRANSIT/ARRIVAL) against the real trade deck and asserts: the dockside step
  renders `MarketFeed` lines derived from `marketAt`/`renderFeed` (not literal fixture text); at
  least one turn's DOCKSIDE step is a `"generate"` step that can fire a real trade-deck incident
  (assert on the resulting `rendered` text or committed fact kind, not on which specific frame
  drew — the draw is seeded but content-agnostic to this test); funds change after a buy and a
  sell action, matching `fundsProjection` over the `purchase.settled`/`sale.settled` facts posted
- snapshot: the new 4-turn script's `announce`-step render keys resolve against real templates
  (extend M1-09's existing template-snapshot suite rather than duplicating it)

## Done when
- a new phase script (JSON, alongside `content/frames/demo/`) drives DOCKSIDE → COMMS → TRANSIT →
  ARRIVAL for at least 4 turns, with each DOCKSIDE (or a clearly-documented equivalent beat) a
  `"generate"` step over `content/decks/trade/frames.json` (falling back to
  `content/decks/generic/frames.json` and the oracle per the existing ladder, unchanged)
- `App.tsx` runs this script via `createPhaseInterpreter(ledger, script, { rng, deck })` instead
  of the M0 demo script; the M0 demo script and its render remain intact and reachable only via
  its own test coverage — do not delete `content/frames/demo/turn.json` or its tests
- the DOCKSIDE screen renders `MarketFeed` fed by `marketAt`/`renderFeed` real projections over
  committed `market.tick` facts (seeded once, same mechanism as the M0 demo's literal-fact setup
  step — INV-6 means the script commits facts, not a click handler computing a live price); the
  script's `purchase.settled`/`sale.settled` amounts are literal but numerically consistent with
  that seeded market data, replacing the M0 demo's arbitrary fixed amount. No new interpreter step
  kind and no live-computed "buy at current price" control — out of scope, see Do-not
- `StatusBar`'s funds field reflects `fundsProjection` changing across the 4-turn playthrough
  (today it only ever shows `0` because no real settlement fact is ever posted in the UI)
- a person can open the built app and click through 4 turns to completion without a console error
  or an unplayable dead end (verify by hand per CLAUDE.md's UI rule, not just by test)

## Do not
- No interrogation-ladder or evidence-reveal overlay (`docs/design/screens-v1.md` §2.2/§2.3) —
  those are follow-on cards; a `"generate"` step firing an incident and rendering its stub text in
  the main panel is sufficient for this card
- No new interpreter step kind and no interactive "buy at live-computed price" control — every
  fact this card commits comes from the script's literal `facts` or the existing `"generate"`
  step, same mechanism the M0 demo already used; do not add a ledger-write path outside the
  interpreter (INV-6) to make a click handler compute a price at runtime
- No confrontation/comms-window resolution — COMMS stays the existing pass-through stub
  (`commsWindow` step kind is already a no-op batch-resolution stub per M2)
- Do not swap the interpreter's `renderSurfaceStub` for M1-09's real `createTemplateRenderer` —
  that is a separate, real gap (the interpreter still stubs `"generate"`/oracle rendering rather
  than calling the real renderer) but conflating the two risks blowing this card's diff past
  CLAUDE.md's ~600-line split point; flag it as a follow-on rather than fixing it here
- No new fact kinds — `purchase.settled`/`sale.settled` already exist and are what
  `fundsProjection` reads; do not invent a `market.trade` kind
- Do not touch `Visibility` handling

## Landed note (2026-07-18)
The "Tests first" snapshot line above was not built as a dedicated snapshot test. Instead,
`content-lint`'s existing semantic check (`step.render && !(step.render in templates)` in
`lint-content.mjs`) already fails the gate if any of `trade-campaign/turn.json`'s render keys
don't resolve to a real template, and `demo-content.test.ts` runs that check as part of `pnpm
test`. That's coverage of the same claim (every render key resolves), just via the existing gate
rather than a new snapshot file — noted here since the card promised a snapshot and this substitutes
for it rather than adding one.

Also touched, not anticipated when this card was written: the M0-09 integration test
(`tests/integration/demo-turn.test.tsx`, renamed to `trade-campaign.test.tsx`) and the e2e smoke
script (`scripts/pages-smoke.mjs`) both asserted the M0 demo's exact button label and copy through
`App.tsx`; swapping `App.tsx`'s script broke both, so both were rewritten to exercise the real
trade campaign instead of the M0 demo. This replaces their M0-era coverage rather than adding
alongside it — `App.tsx` can only run one script at a time, so there was no way to cover both
through the live app. `docs/demos/M0.md` gained a superseded-by note pointing at `docs/demos/M1.md`
for the same reason.
