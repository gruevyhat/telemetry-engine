# BL-05 — `checkIncidentAmbiguity` (INV-10) is never wired in, and several trade-deck frames may not independently satisfy it

## Context
Discovered while building M1-11b's generic-family frames, which the task card requires to
"validate against an adversarial ledger fixture" for ambiguity. Investigating what that actually
means surfaced two related gaps:

1. **`checkIncidentAmbiguity` (`generate/frame.ts`) is never called from any live code path** —
   grep confirms the only reference outside its own definition and its unit test is a comment in
   `validate/validate.ts`. `fireFrame` doesn't call it; `phases/interpreter.ts`'s "generate" step
   doesn't call it. INV-10 ("for every surfaced incident, at least one innocent explanation
   remains consistent with the table-visible fact set") is unit-tested as a standalone function
   but not actually enforced anywhere at runtime today.

2. **Several `content/decks/trade/frames.json` frames (M1-11a) commit only one innocentTwin
   cause fact**, and for cause-fact kinds whose `IMPLIES_V0` clause is entirely `sameActor`-
   correlated (confirmed concretely for `camera.looped` — its one clause requires
   `presence.declared`+`access.granted` both correlated to the *same* actor as the cause fact),
   `consistentActors` structurally cannot return more than the cause fact's own actor, and
   frequently returns an **empty set**, from the frame's own committed facts alone. `trade:
   camera-loop-ghost` is a confirmed case (verified with a scratch test: `consistentActors`
   returns `[]` given only that frame's own cause fact). This would fail
   `checkIncidentAmbiguity` (`actors.size < 2`) if it were ever wired in. Frames whose cause-fact
   kind has a genuinely unconstrained clause (e.g. `lock.cycled`'s `access.granted` correlation,
   which has no `sameActor` at all) are fine *if* the ledger separately contains ≥2
   `access.granted` facts sharing that `codeClass` for different actors — `trade:bay-lock-cycle`
   works in its own unit test only because the test's `ROSTER_FACTS` fixture supplies that
   roster; nothing guarantees an equivalent roster exists in real play.

M1-11b's generic frames sidestep this by having each frame commit **all** the facts its own
ambiguity needs (one primary cause fact + two `access.granted` facts for two different, fixed
actors sharing one codeClass) — self-contained, provable against a minimal/adversarial ledger
with nothing pre-existing. The trade deck's frames were not built this way and weren't required
to be by M1-11a's own card.

## Tests first
- wire `checkIncidentAmbiguity` (or an equivalent gate) into the real fire path (`fireFrame`, or
  the interpreter's "generate" step, or a dedicated validator pass) and add a test proving a
  frame that fails ambiguity is rejected/flagged, not silently committed
- for each `content/decks/trade/frames.json` frame, a test asserting `checkIncidentAmbiguity`
  passes given *only* that frame's own committed facts (the adversarial-fixture standard M1-11b
  uses) — expect `trade:camera-loop-ghost` (and possibly others using single-cause-fact,
  self-consistency-only implies clauses) to need a similar self-contained fix: either an
  additional supporting cause fact (mirroring the generic frames' pattern), or a different
  cause-fact kind whose implies clause is genuinely unconstrained

## Done when
- INV-10 is enforced on a real, reachable path, not just unit-tested in isolation
- every shipped trade-deck frame independently satisfies ambiguity against a minimal ledger
  containing only its own committed facts

## Do not
- Do not weaken or remove `checkIncidentAmbiguity`'s existing unit tests
- Do not change `IMPLIES_V0`'s existing entries without checking every frame that currently
  relies on them
