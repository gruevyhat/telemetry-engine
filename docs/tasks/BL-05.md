# BL-05 — the inference bot proves ambiguity in a test suite, but nothing enforces it on a live/reachable path

## Context
Originally discovered while building M1-11b's generic-family frames (needing to "validate
against an adversarial ledger fixture" for ambiguity), and resolved for content during M1-12:

- **M1-12 built the real inference bot** (`packages/engine/src/inference/bot.ts`'s
  `consistentWorlds`) — roster enumeration filtered by the position model and (when one exists)
  `IMPLIES_V0` closure, per fact-kinds-v0.md §3's definition. `packages/content-lint/src/
  inv10-property.test.ts` runs it against every frame in both shipped decks (trade, generic) as a
  zero-tolerance property test.
- **That property test found and this milestone fixed three real trade-deck violations**:
  `trade:bay-lock-cycle` and `trade:manifest-shortfall` committed only their primary cause fact
  with no supporting facts for their own `IMPLIES_V0` clause to enumerate over; `trade:
  camera-loop-ghost` used `camera.looped`, whose one clause is entirely `sameActor`-correlated and
  structurally cannot enumerate ≥2 actors regardless of what's committed (fixed by changing its
  cause-fact kind to `lock.cycled`, not by adding facts). All frames in both decks now pass.
- **What's still open**: `consistentWorlds`/`checkIncidentAmbiguity` are proven correct and
  exercised against real content, but **nothing calls either of them from a live/reachable game
  path** — `fireFrame` doesn't call them, `phases/interpreter.ts`'s "generate" step doesn't call
  them. A future frame authored without running (or without correctly reading the output of) the
  content-lint property suite could still ship a real ambiguity violation and nothing at runtime
  would catch it before it fires at a real table.

## Tests first
- wire `consistentWorlds` (or an equivalent gate) into a real, reachable path — the composer that
  eventually selects/fires frames (once one exists beyond frame-by-id `generate` steps), or a
  content-lint-time check that's part of the required gate rather than an opt-in test file, or
  both — and add a test proving a frame that fails ambiguity is rejected/flagged, not silently
  fired

## Done when
- INV-10 is enforced on a real, reachable path, not just proven in a property test a developer
  has to remember to run

## Do not
- Do not weaken or remove `consistentWorlds`/`checkIncidentAmbiguity`'s existing unit/property
  tests
- Do not change `IMPLIES_V0`'s existing entries without checking every frame that currently
  relies on them
