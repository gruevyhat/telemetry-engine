# M1-15 — NPC interrogation ladder UI

## Context

`docs/design/screens-v1.md` §2.2 designed this screen for M1 but no card ever built it; M1-13
scoped it out explicitly to stay under the diff-size split point. Unlike M1-13's dockside
market/incident wiring, this is a **player-initiated, mid-beat action** (Persuade/Intimidate on an
NPC, reachable during COMMS or DOCKSIDE) — it cannot be pre-scripted as a script's literal `facts`
the way M1-13's buy/sell were, because the player chooses when and whom to interrogate, and the
resulting `check.reported` fact and the NPC's ladder-tiered answer depend on that live choice.

INV-6 ("nothing writes to the ledger except the phase-engine interpreter") means only
`createPhaseInterpreter`'s `advance()` may append facts. Today's interpreter has no API for "commit
a player-chosen action inside the current beat" — `queueCommsAction` exists but is an inert M2
stub (`commsWindow` step kind is a no-op batch-resolution stub). Two materially different designs
both fit the Spec's silence on this:
1. Model interrogation as its own `branch`+`check` sub-sequence of script steps (stays inside the
   existing step-kind vocabulary, but a script can't express "any NPC, chosen at play time" without
   either enumerating every NPC as separate steps or a templated/generated sub-script).
2. Add a new interpreter action API (e.g. an `interrogate(actor, npcId, approach, checkTotal)`
   method alongside `advance`) that appends its own transaction directly, still gated through the
   interpreter (INV-6 preserved) but a new surface distinct from `advance`/`StepInput`.

## Owner decision (2026-07-18)

Option 2 (new interpreter action API alongside `advance`) is approved. Implemented as
`PhaseInterpreter.reportCheck(t, actor, { skill, dm, total, difficulty })`, which computes
`effect = total - difficulty` and appends exactly one `check.reported` fact through the same
`ledger.append` call site `advance()` uses (INV-6 preserved: still the interpreter, and only the
interpreter, writing). It does not advance `currentStep()` — a side action, not a beat transition.

Note for future action APIs (evidence, M1-16): `reportCheck` does not itself widen any fact's
visibility — the interrogation answer it feeds (`assembleInterrogationAnswer`) only *selects among
an NPC's own already-committed facts* to decide what the rendered answer says; it never mutates or
reveals those facts' `Visibility`, so nothing here touches Visibility handling. Evidence's reveal
mechanism is different in kind (it does widen scope) and needs its own decision — see M1-16.

## Tests first
- `interpreter.test.ts`: `reportCheck` commits exactly one `check.reported` fact with the correct
  payload (`actor`, `skill`, `dm`, `total`, `difficulty`, `effect`), `public` visibility per
  `kinds-v0.ts`, and does not change `currentStep()`
- `App.test.tsx`: an "Interrogate" control (Persuade/Intimidate) is reachable during COMMS; a
  submitted roll renders an answer whose content matches `assembleInterrogationAnswer`'s truth-tier
  bucket for the entered effect (evasion/partial/trueWithTell/true), and the answer is presentation
  text only — assert no new ledger fact besides `check.reported` results from it

## Done when
- `PhaseInterpreter` gains `reportCheck` (engine change, `packages/engine/src/phases/
  interpreter.ts`), covered by the tests above
- `App.tsx` renders an interrogation control at COMMS targeting the NPC named in whichever
  incident fired that turn (read from the committed cause fact's `actor`, not hardcoded), calls
  `reportCheck`, then renders `assembleInterrogationAnswer`'s result as presentation text
  (INV-12: never written back as a fact)

## Do not
- Do not build this without the owner decision above recorded in this card first
- Do not add free-text question entry — Persuade/Intimidate are the only two actions per Spec §12
- Do not touch `Visibility` handling
