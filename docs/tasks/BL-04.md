# BL-04 â€” `pnpm typecheck` only checks ui-shared and ui-phone, not the other five packages

## Context
Discovered while adding a type-level test to `packages/engine/src/render/renderer.test.ts`
(M1-09): `package.json`'s `typecheck` script has only ever run
`tsc --noEmit -p packages/ui-shared/tsconfig.json && tsc --noEmit -p packages/ui-phone/tsconfig.json`.
Neither ui-shared's nor ui-phone's tsconfig references `packages/engine` via TS project
references, so **no gate has ever type-checked `packages/engine`'s own source or test files** â€”
every type error in the engine since M0 would have had to be caught by `vitest run`'s esbuild
transform (which does not type-check at all) or a reviewer's eyes, not by `pnpm typecheck`.

`packages/engine`, `packages/plugin-stub`, and `packages/plugin-traveller`
each have a `tsconfig.json` that already `tsc --noEmit`s clean today. `packages/content-lint`
does not: its one test file uses `node:child_process`/`node:url`/`import.meta.url`/`process`
without a Node types setup, which is a real, separate gap (missing `@types/node` or an
equivalent lib/types config) â€” fixing it needs a new dependency, which is a stop-and-ask
condition per CLAUDE.md, not something to guess at while fixing the typecheck script's coverage.

M1-09's own green commit adds `packages/engine`'s tsconfig to the `typecheck` script (needed to
make that task's own `@ts-expect-error` type-level test meaningful), but stopped there rather
than also sweeping in plugin-stub/plugin-traveller/sim/content-lint, since only `engine` was
required by that task.

**Update (M1-12):** `packages/sim` no longer belongs in the "already clean" list above â€” its
skeleton (`export {}`) never exercised this. Once M1-12 gave it real modules that (a) use
`node:fs`/`process`/`import.meta.url` (content-lint's exact missing-`@types/node` gap) and (b)
import `packages/engine/src/**` across the package boundary the way content-lint's *tests* do,
`tsc --noEmit -p packages/sim/tsconfig.json` now fails with both the Node-types errors and a
second, distinct class: `TS6059`/`TS6307` ("File X is not under rootDir" / "not listed within the
file list of project Y") on every cross-package import, because neither tsconfig declares a
`references` entry to the other and `rootDir`/`composite` assume a project only contains its own
files. `vitest` doesn't hit this because its on-the-fly esbuild transform never runs a full
project-wide `tsc` type-check â€” it explains why this shipped fully green through `pnpm test` /
`pnpm lint` / `pnpm typecheck` (the last of which never looked at `packages/sim` either). Fixing
this for real needs either genuine TS project references (`sim`'s tsconfig referencing `engine`'s,
mirroring `build-stub.mjs`'s existing `tsc --build` precedent) or a resolution-mode change; either
is more than a one-line script edit, so it's left for whoever picks up this card rather than
guessed at here.

## Tests first
- none language-level; this is a script/config fix. Verify by running `pnpm typecheck` before and
  after and confirming it now fails on a deliberately-introduced type error in each newly-covered
  package (plugin-stub, plugin-traveller, sim), then reverting the deliberate error.

## Done when
- `typecheck` in `package.json` also runs `tsc --noEmit` over `packages/plugin-stub`,
  `packages/plugin-traveller`, and `packages/sim`'s tsconfigs
- `content-lint`'s Node-types gap is fixed (with owner sign-off on whatever dependency that
  needs) and it is added to the script too, OR a decision is recorded that content-lint is
  intentionally excluded and why

## Do not
- Do not add a new dependency (e.g. `@types/node`) without owner sign-off first
