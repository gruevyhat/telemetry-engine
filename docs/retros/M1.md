## What the gate caught / what it missed

**Gate catches, working as intended:** M1-12's `inv10-property.test.ts`, run for the first time
against real content, found three genuine INV-10 (an incident with a real innocent twin must
never uniquely attribute to one actor) ambiguity violations in the trade deck — fixed in content
(`b5bb982`), not by weakening the check. This is the gate doing exactly its job: a property test
written early (M1-05) caught a real defect only once real content existed to run it against
(M1-12).

**Gate misses — defects the automated gate could not have caught, found only by review or
after-the-fact:**

1. **INV-6 violation** (nothing writes to the ledger except the phase-engine interpreter):
   `commitEvidenceReveal`/`commitInterrogationAnswer` called `ledger.append`/`appendAll` directly
   from outside `packages/engine/src/phases/`. `eslint-rules/no-ledger-writes-outside-interpreter.js`
   didn't catch it — it's a syntactic name-matcher that doesn't pattern-match `.appendAll`, a known
   gap already filed as `docs/tasks/BL-03.md` at the M0 retro and still open. Caught by review
   (`2f8e82d`, properly documented after the fact in `1449563` since the original commit message
   undersold what it actually fixed), not any gate.
2. **INV-1 violation** (`packages/engine` must never import from `content/`): a trade-deck smoke
   test imported a content JSON file directly from inside `packages/engine`. No lint rule enforces
   this import ban; `build:stub`'s INV-1 check only proves `plugin-traveller` is excludable from
   the build, and doesn't compile test files, so it was answering a different question. Caught by
   review same day (`d4acbdb`).
3. **INV-2/3 divergence** (replay determinism, including across a kill/resume): the oracle's RNG
   stream was scoped to the interpreter instance's lifetime rather than derived from the ledger,
   so a killed-and-resumed process would draw differently than an uninterrupted run on its next
   oracle/generate draw. Caught by a new kill-resume replay test written specifically to probe for
   this (`d0d7108`), fixed by re-scoping every step's stream to a ledger-derived visit count
   (`9686417`) — this one *was* caught by a test, but the test had to be deliberately added; nothing
   in the existing suite exercised kill/resume before this.
4. **A bundled, mislabeled commit** (M1-08): the interrogation-ladder implementation and a real
   INV-6 fix landed inside a commit (`2f8e82d`) whose message describes only two unrelated
   test-expectation corrections — self-corrected same day (`1449563`) once noticed, but nothing
   mechanical would have caught the mislabeling itself.
5. **Two commits not red-first** (M1-15's UI wiring, `2a01bbe`/`219e11b`): `App.test.tsx` and
   `App.tsx` changes landed together with no preceding failing-test commit, a real deviation from
   CLAUDE.md's TDD-loop rule. Found by `git show --stat` review, not any gate — nothing enforces
   red-before-green mechanically in this repo.
6. **The sim harness's `recurrenceRate` metric was wrong by ~27x** (found and fixed during this
   PR's review, after the session that landed M1-13–16, on `milestone/M1` directly): `export.ts`
   flattened every campaign's events into one array before computing recurrence, but the metric
   compares turn numbers with no campaign identifier — so a frame firing in campaign 1 and again
   in campaign 7 counted as a recurrence. Published number: 0.95 against Spec §21.4's <5% target.
   Real number, fixed: L1 3.5%, L2 7%, L4 4%. No test exercised more than one campaign sharing a
   frame at the same turn number before this — `export.test.ts`'s existing two-campaign case
   happened to use disjoint frame IDs. Fixed with a red/green pair (`6cbcf3d`/`c6e0959`).
7. **CI gate coverage is thinner than green checkmarks suggest, and predates M1**: the `deploy`
   job (the only place `build:pages`/`test:e2e` run) is gated to pushes on `main`, skipping every
   PR. `content-gate` reported "pass" on PR #8 while skipping every step — its change-detection
   evaluated false on the pull-request event, so `lint:content`/`sim:smoke` never ran despite the
   PR adding substantial `content/`. `test:integration` runs in no CI job at all. M0's PR #5 merged
   under the same `deploy` gating, so this isn't new to M1, but the `content-gate` false-skip and
   the missing `test:integration` wiring hadn't been surfaced before this review.

Common thread, same as M0's retro found: every miss above is something the gate couldn't observe
about itself — an import path no lint rule parses, an RNG lifetime no earlier test exercised, a
commit message that doesn't match its diff, a metric aggregation bug with no adversarial test
case, a CI conditional never run against the event shape that actually matters (pull_request, not
push). None were caught by "the gate"; all were caught by someone deliberately checking the gate's
own blind spots.

## Fun check

The owner ran the full by-hand walkthrough on 2026-07-18 (`docs/demos/M1.md`'s Run record) and
confirmed **mechanical completion**: all 20 advances, the check/interrogation/evidence controls,
no console errors. The **fun assessment failed** — recorded verbatim: "the owner could not imagine
how the current loop would be fun." That is the entire honest record available from this session;
nothing here should be read as this implementer session's guess at *why* it wasn't fun, since that
requires the owner's own account of what dragged, which hasn't been captured yet. This retro
surfaces one adjacent data point that might be relevant to that conversation without asserting a
causal claim: recurrence within a playthrough is genuinely low (3.5–7%, see Sim/quality trend
above), so if the "why" investigation starts from "the incidents feel repetitive," the measured
data doesn't support that as the mechanism — worth ruling out early rather than spending tuning
effort there.
