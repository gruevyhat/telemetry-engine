## What the gate caught / what it missed

**Red-phase catches (the large majority, working as intended):** every task's red commit failed
for its stated reason and was confirmed via `pnpm test` before implementation — M0-01's lint
rules, M0-02's ledger/registry properties, M0-03's reducer/projection stubs, M0-05's RNG
stream-independence, M0-06's phase-load validation, M0-08's persistence round-trip, M0-09's demo
rendering, M0-10's deploy-contract checks, and later the orphaned-preview regression and the CI
content-gate push-detection test. TDD discipline held throughout; no task skipped red.

**Gate misses — defects the automated gate could not have caught, found only by manual work or
after-the-fact review:**
1. **The UI packages (`ui-shared`, `ui-phone`) have no typecheck step at all.** `pnpm build:stub`
   *does* run a real typecheck for the engine (`scripts/build-stub.mjs` runs `tsc --build` on
   `packages/engine` + `packages/plugin-stub`) — that part of the gate is sound. But
   `ui-shared`'s and `ui-phone`'s `build` scripts are just `vite build`, and `pnpm test` runs
   through Vitest's esbuild transform, which strips TypeScript types without checking them. M0-01's
   missing-`@types/react`/`@types/react-dom` gap (PR #1) was in exactly these UI packages, and
   `pnpm test`'s first green run masked it because esbuild never checked the type at all. This
   hole was open for the entire milestone — no command anywhere typechecked `ui-shared`/
   `ui-phone` — and is closed by Action 1 below. (PR #2 also describes a cast that "didn't compile
   under `tsc --noEmit`" and calls it a `build:stub` miss; that one is inside the engine, which
   `build:stub` does typecheck, so that specific claim looks inaccurate on inspection — logged here
   rather than repeated as fact.)
2. **The CI content-gate's own trigger logic was silently broken for the whole milestone.**
   `content-gate` diffed `origin/$BASE_REF...HEAD` unconditionally; on a push to `main`,
   `origin/main` already points at the pushed HEAD, so that range is always empty and
   `lint:content`/`sim:smoke` silently skipped on exactly the branch that deploys. No test existed
   that checked the gate's conditional logic against real push-event semantics — `deploy-workflow.
   test.js` checked string content of the YAML, not what it does on the events it actually
   receives. Only caught this session, via an explicit historical review, not by any milestone
   gate. Fixed in `a9a4400`/`77cb7ca`, with a regression test now in place.
3. **Locale-dependent funds rendering.** `StatusBar` used `funds.toLocaleString()` with no locale
   argument, so the render followed the host machine's locale (`Cr169,200` vs. `Cr169.200`). CI
   always runs in a fixed (en-US-equivalent) locale, so this could never go red in the normal
   pipeline — it was found by deliberately running the test under `LC_ALL=de_DE.UTF-8`, prompted
   by Spec §21.3's "replays identically on two machines" requirement, not by any gate. Fixed in
   `347f8ea`. **Owner decision (2026-07-13): the locale pin plus the existing replay-determinism
   property tests satisfy §21.3 without a literal second-machine run** — closed, see
   `docs/demos/M0.md`'s Run record.
4. **Orphaned Vite preview process after `pnpm test:e2e`.** The smoke script only signaled the
   `pnpm` wrapper it spawned, not the Vite process underneath, so the port stayed bound after
   teardown. CI only runs the e2e job once per run, so a single-run gate would never notice a
   leftover process; this was found by an implementer actually running `pnpm demo:m0` after
   `pnpm test:e2e` and hitting a bound-port failure. Fixed in `5130d6b` with real process-group
   signaling, verified by two consecutive runs.

Common thread: every gate miss above is a case where the gate can't observe its own blind spot —
a typecheck no command runs, a conditional no test exercised against real event shapes, a
locale/process-lifecycle property that only differs outside the CI runner's own environment. None
were caught by "the gate", all were caught by someone manually doing the thing the gate was
supposed to stand in for.

## Spec health

No Spec amendment PRs were opened during M0. Seven extrapolation notes from M0-01–M0-10 commit
messages and PR bodies were untriaged into Spec §24 (Open Questions) or a resolved amendment, per
Plan §7's "triaged at each retro" rule. Owner-triaged in this retro:

1. `npc.statement`'s `truthTier` companion fact kind is named in prose (`fact-kinds-v0.md`) but
   never given a kind name — `npc.statement` shipped as `{npcId, topic}` only (PR #2). **Owner
   decision: open a catalog PR now.** Landed in this retro — see `fact-kinds-v0.md`.
2. v0 catalog payload field *types* (as opposed to names) were all inferred from surrounding prose,
   not specified anywhere (PR #2). **Owner decision: amend the catalog now.** Landed in this
   retro — every kind's payload fields now carry explicit types.
3. `presence.declared`'s "station or hex, exactly one" was modeled as two independent optional
   fields with no enforced exclusivity (PR #2). **Owner decision: fix it now.** Landed in this
   retro as a real registry-level constraint, with a red/green test pair.
4. `clock.tick`'s visibility is "per clock definition" per the catalog, but the registry can only
   express one static default; it's currently defaulted to `'referee'` pending the real
   per-clock-def mechanism. **Owner decision: log as an open question, owned by M1-01** (added to
   Spec §24).
5. `DifficultyRef` (used by the phase interpreter's `check` step kind) has no Spec definition
   anywhere and is currently just `number` (M0-06 commit `20f4133`). **Owner decision: log as an
   open question** (added to Spec §24).
6. `PhaseStep`'s Spec-shown interface had no field for `tickClock`'s config; the implementation
   added `tick?: {clockId, delta}` (M0-06). **Owner decision: patch the Spec diagram directly** —
   landed in this retro (Spec §4).
7. `no-ledger-writes-outside-interpreter` (the INV-6 lint rule) is a v0 syntactic matcher — it
   flags an identifier literally named `ledger` under a `phases/` path, not a type-checked taint
   rule, because the real Ledger type didn't exist yet when M0-01 wrote it (PR #1, extrapolation
   #2). It was never revisited once M0-02 landed the real Ledger API. **Owner decision: file a
   task card** — see `docs/tasks/BL-03.md`.

Sections implementers repeatedly had to reason carefully about rather than just read off the page:
the beat-sequence ownership question (M0-04's commit explains at length why the engine must *not*
hard-code DOCKSIDE→COMMS→TRANSIT→ARRIVAL, reading Spec §4's Do-not against the task card's "beat
order enforced" wording) and the "closed cycle vs. terminal step" reading of phase scripts (M0-06).
Neither caused a wrong implementation, but both took real interpretive work — possible rewrite
candidates if M1 authors hit the same ambiguity from a colder start.
